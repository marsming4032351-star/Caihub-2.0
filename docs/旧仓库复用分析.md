# 旧仓库复用分析

分析对象：`marsming4032351-star/Caihub-2.0`

## 结论

旧仓库有明确复用价值，但价值主要在以下层面：

- 平台概念建模
- 数据资产定义
- 数据契约思想
- Agent / Skill 编排方法
- 事件驱动的数据流设计

它更像“架构知识库 + 产品蓝图 + 治理文档库”，不是一套可直接迁移的后端实现。

## 可以复用的高价值内容

### 1. 事件优先视角

旧仓库最重要的一点不是“识别菜品”，而是“把一次真实出品沉淀为结构化事件”。

这对当前项目的启发是：

- 不能只保留 `dish` 主数据
- 必须补 `production event` 契约
- 后续质检、分析、训练都应围绕事件展开

### 2. 多模态采集

旧仓库强调：

- 图像
- 温度
- 重量
- 时间

共同构成有效输入。

这说明当前项目的 `vision` 模块不能长期只停留在单图识别，后续必须升级为多模态输入模型。

### 3. 模型与规则分离

旧仓库把“识别”和“裁决”拆成两层：

- 模型负责识别与对齐
- 规则引擎负责合格性判断

这对当前项目很重要，因为这能避免把业务判断硬编码进模型输出。

### 4. Skill Marketplace 思路

旧仓库明确提出：

- Agent 是编排者
- Skill 是能力商品

这意味着当前仓库中的 `agents` / `skills` 目录不应只是说明性存在，后续要逐步演化为可注册、可审计、可组合的能力中心。

### 5. 数据契约和仪表盘意识

旧仓库中的契约规范和仪表盘设计说明，后续 CaiHub 不只是服务接口，还会是数据产品平台。

## 不建议直接照搬的部分

### 1. Obsidian / Dataview 组织方式

旧仓库大量内容围绕 Obsidian 知识库组织，这适合前期知识沉淀，不适合作为运行时后端的主结构。

### 2. 过强的文档中心结构

旧仓库目录很完整，但更偏战略与方法论文档，对当前代码仓来说太重。应提炼原则，而不是照搬十几个顶层编号目录。

### 3. 部分能力仍停留在概念层

例如：

- Skill Marketplace
- 多 Agent 编排
- 全链路规则裁决

这些方向是对的，但当前代码仓应先做“契约 + 事件 + 服务边界”的最小落地。

## 已经吸收进当前项目的内容

本轮调整已经吸收了以下旧仓库价值：

- 新增 `production` 领域契约
- 在系统蓝图中增加“事件优先、多模态、规则分离、数据资产化”核心原则
- 增加品控裁决代理和多模态采集 / 规则引擎技能
- 在数据流中补充“出品事件 -> 规则引擎 / 历史沉淀”的链路

## 后续建议

按优先级建议这样推进：

1. 建立 `production_event` 持久化模型和接口
2. 为 `quality rule engine` 建立显式 service
3. 为契约增加兼容性测试和 schema 导出
4. 再考虑真正的 agent 编排运行时
