# CaiHub 业务版项目说明

本文档面向业务合作方、管理层和投资人，用于快速理解 CaiHub 当前在做什么、为什么有价值、现在走到哪一步。

## 1. CaiHub 是什么

CaiHub 不是一个单纯“识别菜品图片”的工具。

它的目标是把餐饮场景中的每一次真实出品，转化为：

- 可识别
- 可判断
- 可追溯
- 可沉淀
- 可复用

的数据资产。

一句话概括：

```text
CaiHub 要把“后厨经验”和“菜品标准”变成可计算、可治理、可规模复制的平台能力。
```

## 2. 解决什么问题

餐饮行业常见问题包括：

- 同一道菜，不同厨师出品波动大
- 标准难传承，培训成本高
- 门店质量难实时判断
- 图片、经验、记录分散，无法形成长期资产

CaiHub 试图解决的不是“看懂一张图”，而是：

```text
如何让一次出品有标准、有判断、有记录、有回流
```

## 3. 当前核心思路

CaiHub 当前采用的是一条“标准驱动的事件链路”：

```text
菜品定义
   ↓
菜品标准
   ↓
真实出品事件
   ↓
质量裁决
   ↓
数据沉淀
   ↓
后续分析 / 训练 / 优化
```

这里的关键不是“识别”，而是“裁决”。

也就是说：

- 模型负责看懂
- 标准负责约束
- 规则负责判断
- 数据负责沉淀

## 4. 当前平台结构

从业务视角看，当前平台已经有四个基础模块：

### 1. 菜品资产

管理菜品本身的主数据，例如：

- 名称
- 描述
- 价格
- 可售状态

### 2. 菜品标准

定义一道菜应该达到什么标准，例如：

- 目标重量
- 温度范围
- 最低识别置信度
- 最大允许偏差

### 3. 出品事件

记录一次真实出品，包括：

- 哪家门店
- 哪位操作人
- 图像
- 温度
- 重量
- 识别结果
- 最终裁决

### 4. 系统蓝图

维护平台自身的治理信息，包括：

- 数据契约
- 数据流
- agent
- skill

这意味着 CaiHub 当前已经不只是接口集合，而是在向“平台型仓库”演进。

## 5. 当前差异化价值

### 1. 不是只做视觉识别

多数方案停留在“识别菜像不像”。

CaiHub 更关注：

- 这次出品是否合格
- 为什么不合格
- 是否可以沉淀成长期资产

### 2. 不是只做模型

CaiHub 当前设计中明确把三件事拆开：

- 模型识别
- 标准定义
- 规则裁决

这意味着平台不会把全部能力绑死在某一个模型上。

### 3. 不是只做单店工具

项目结构已经提前考虑：

- 数据契约
- 数据流
- 多领域拆分
- agent / skill 扩展

这更接近“平台能力底座”，而不是一次性项目。

## 6. 当前已经完成到什么程度

当前项目已经完成第一阶段基础建设，具备：

- 可运行的后端服务
- 菜品接口
- 菜品标准接口
- 出品事件接口
- 基于标准的最小质检规则
- 视觉识别接口
- 数据契约与数据流骨架
- 数据库迁移基础设施
- 面向后续 agent / skill 扩展的结构预留

这说明项目已经从“概念设计”进入“可验证原型”阶段。

## 7. 当前还没做完的部分

当前还没有完全进入大规模业务化，主要还缺：

- 门店实体
- 操作人实体
- 更强的规则引擎
- 真实事件总线
- 数据分析导出
- agent 运行时编排

所以当前阶段更适合定义为：

```text
平台基础层完成，业务深水区尚未展开
```

## 8. 为什么当前阶段值得继续推进

因为现在已经把最难统一的基础边界先搭起来了：

- 数据对象边界
- 标准边界
- 事件边界
- 契约边界
- 治理边界

这些边界一旦前期做错，后面系统会非常难扩。

当前先把这些边界搭好，后续无论是：

- 做门店质检
- 做 SOP 培训
- 做菜品知识库
- 做品牌标准化输出
- 做 agent 协作

都会更稳。

## 9. 下一阶段最值得投入的方向

从业务价值和工程顺序一起看，建议下一阶段重点投入：

1. 门店与人员体系
   让出品事件不再只是字符串记录，而是进入真实组织结构。

2. 规则引擎升级
   把当前最小规则逻辑升级成更强的标准判定系统。

3. 数据回流与分析
   让出品历史真正变成报表、训练样本和标准优化依据。

4. Agent / Skill 落地
   把现在的注册层推进成真正的自动化协作能力。

## 10. 一句话总结

```text
CaiHub 当前不是一个“识别 demo”，而是在搭一个以菜品标准、出品事件和数据资产为核心的餐饮智能平台底座。
```
