## 2. 代码分层图

```text
app/
├── api/             对外暴露 HTTP 接口
├── core/            应用工厂、配置、生命周期
├── db/              会话、元数据、模型注册
├── models/          ORM 持久化模型
├── schemas/         请求和响应模型
├── repositories/    数据读写
├── services/        业务逻辑
├── domains/         数据契约
├── mesh/            Data Mesh 注册表
├── events/          领域事件
├── agents/          Agent 注册表
├── skills/          Skill 注册表
└── vision/          视觉算法与识别
```

### 分层职责

- `api`
  只负责路由、依赖注入、HTTP 协议边界。
- `services`
  负责业务逻辑和跨对象编排。
- `repositories`
  负责数据库访问。
- `models`
  负责数据库结构表达。
- `schemas`
  负责 API 输入输出。
- `domains`
  负责数据契约定义。
- `mesh`
  负责平台级数据产品元信息。
- `events`
  负责事件定义，为异步化预留边界。

## 8. 当前阶段定位

当前 CaiHub 仓库的定位不是“完整业务系统”，而是：

- 可运行的 FastAPI 后端基础仓
- 面向 Data Mesh 的领域化后端骨架
- 面向数据契约和数据流治理的平台仓库
- 面向 agent / skill 扩展的基础设施仓库
