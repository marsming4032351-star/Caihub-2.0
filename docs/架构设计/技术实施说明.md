# CaiHub AI Company 技术实施说明

本文档面向开发团队，说明当前仓库如何支撑 CaiHub 的 AI Company 方向，以及后续扩展时如何保持架构边界不被破坏。

## 1. 当前技术栈

```text
API 框架      FastAPI
ORM          SQLAlchemy 2.x
配置管理      Pydantic Settings
迁移工具      Alembic
数据库        PostgreSQL（默认）/ SQLite（测试）
视觉能力      OpenCV + NumPy
测试          Pytest
```

## 2. 当前代码模块职责

### `app/api`
- 定义 HTTP 路由
- 管理 FastAPI 依赖注入
- 作为外部调用边界

### `app/services`
- 编排业务流程
- 提供系统蓝图和架构元数据
- 作为业务服务与未来 Agent runtime 的衔接层

### `app/repositories`
- 封装数据库读写
- 屏蔽 SQLAlchemy 细节

### `app/models`
- 定义持久化模型
- 反映数据库结构

### `app/schemas`
- 定义 API 输入输出模型
- 定义系统蓝图返回模型

### `app/domains`
- 定义领域级数据契约
- 服务 Data Mesh 与数据资产治理

### `app/mesh`
- 维护数据产品清单
- 维护数据流清单
- 为系统蓝图接口提供统一元数据

### `app/events`
- 定义领域事件
- 为真实事件总线预留边界

### `app/agents` / `app/skills`
- 维护平台级 Agent / Skill 注册表
- 当前是治理与元数据层，不是最终 runtime

### `app/vision`
- 承接当前视觉识别能力
- 未来对接硬件层和质量判断链路

## 3. 当前关键调用链

### 3.1 菜品接口

```text
POST /api/v1/dishes
  -> DishService
  -> DishRepository
  -> dishes 表
```

### 3.2 菜品标准接口

```text
POST /api/v1/standards
  -> DishStandardService
  -> DishStandardRepository
  -> dish_standards 表
```

### 3.3 出品事件主链路

```text
POST /api/v1/production/events
  -> ProductionEventService
      -> DishStandardService 查标准
      -> QualityRuleService 做裁决
      -> ProductionEventRepository 落库
      -> quality_decision_event 生成领域事件
  -> production_events 表
```

### 3.4 系统蓝图接口

```text
GET /api/v1/system/architecture
  -> ArchitectureService
      -> mesh.contracts
      -> mesh.flows
      -> agents.registry
      -> skills.registry
```

## 4. 当前仓库和最终架构的差距

### 当前已经实现
- 平台基础 API
- 菜品 / 标准 / 出品事件主链路
- 最小质量裁决闭环
- 视觉识别接口
- 数据契约与数据流元数据注册
- Agent / Skill 注册治理

### 还未真正实现
- CEO Agent 编排中心
- Vision QA / Menu R&D / Store Ops / Marketing / Data Asset 的执行 runtime
- 硬件层适配器
- 数据资产闭环分析与导出
- 真实事件总线

## 5. 扩展开发顺序

建议按下面顺序推进：

```text
第一层
  store
  operator
  rule_engine
  agent runtime skeleton

第二层
  event_bus
  analytics_export
  hardware adapter
  contract compatibility test

第三层
  CEO Agent orchestration
  Data Asset feedback loop
  API ecosystem
```

## 6. 开发边界约束

### 新增一个业务对象时
至少补齐：

```text
model
schema
repository
service
api endpoint
test
如涉及稳定输出，再补 contract
如涉及数据流，再补 mesh metadata
```

### 新增一个稳定数据产品时
至少补齐：

```text
app/domains/<domain>/contracts/
app/mesh/contracts.py
app/mesh/flows.py
tests
相关 docs
```

### 新增一个 Agent 或 Skill 时
至少补齐：

```text
app/agents/registry.py
app/skills/registry.py
docs/智能体/
docs/技能/
必要时补 system/architecture 测试
```
