# CaiHub 技术实施说明

本文档面向开发团队，说明当前仓库如何运行、如何扩展、如何保持架构边界不被破坏。

## 1. 当前技术栈

```text
API 框架      FastAPI
ORM          SQLAlchemy 2.x
配置管理      Pydantic Settings
迁移工具      Alembic
数据库        PostgreSQL（默认）/ SQLite（测试）
视觉能力      OpenCV + NumPy
测试          Pytest
```

## 2. 当前模块职责

### `app/api`

职责：

- 定义 HTTP 路由
- 管理 FastAPI 依赖注入
- 作为外部调用边界

约束：

- 不直接写数据库查询
- 不直接写业务规则
- 不直接组织跨领域流程

### `app/services`

职责：

- 编排业务流程
- 调用 repository
- 组合多个领域对象
- 输出 API 需要的结果

约束：

- 可以包含业务判断
- 不应包含数据库底层访问细节

### `app/repositories`

职责：

- 负责数据库读取和写入
- 屏蔽 SQLAlchemy 查询细节

约束：

- 不承担业务规则
- 不处理 HTTP 请求语义

### `app/models`

职责：

- 定义持久化模型
- 反映数据库表结构

### `app/schemas`

职责：

- 定义 API 输入输出模型
- 定义系统蓝图返回结构

注意：

- `schemas` 不等于数据契约
- 数据契约在 `app/domains/*/contracts`

### `app/domains`

职责：

- 定义领域级数据契约
- 服务 Data Mesh 治理
- 为未来事件、导出、分析提供稳定 schema

### `app/mesh`

职责：

- 维护数据产品清单
- 维护数据流清单
- 为系统蓝图接口提供统一元数据

### `app/events`

职责：

- 定义领域事件信封
- 为后续事件总线预留边界

当前状态：

- 已有事件定义
- 还没有真实消息中间件集成

### `app/agents` / `app/skills`

职责：

- 维护平台级 agent / skill 注册表
- 为后续自动化协作和治理提供基础清单

当前状态：

- 当前是注册层
- 还不是可执行调度运行时

## 3. 当前关键调用链

### 3.1 菜品接口

```text
POST /api/v1/dishes
  -> DishService
  -> DishRepository
  -> dishes 表
```

### 3.2 菜品标准接口

```text
POST /api/v1/standards
  -> DishStandardService
  -> DishStandardRepository
  -> dish_standards 表
```

### 3.3 出品事件主链路

```text
POST /api/v1/production/events
  -> ProductionEventService
      -> DishStandardService 查标准
      -> QualityRuleService 做裁决
      -> ProductionEventRepository 落库
      -> quality_decision_event 生成领域事件
  -> production_events 表
```

### 3.4 系统蓝图接口

```text
GET /api/v1/system/architecture
  -> ArchitectureService
      -> mesh.contracts
      -> mesh.flows
      -> agents.registry
      -> skills.registry
```

## 4. 当前数据库对象

```text
dishes
dish_standards
production_events
```

### 设计说明

- `dishes`
  保存菜品主数据
- `dish_standards`
  保存按 `dish_key` 管理的标准定义
- `production_events`
  保存真实出品事件和质量裁决结果

### 当前不足

- `dish_id` 仍是字符串，而不是外键关系
- `store_id` 和 `operator_id` 也还是字符串

这意味着当前是“可运行的平台早期形态”，不是最终关系模型。

## 5. 迁移与启动规范

### 正式方式

```bash
export CAIHUB_DATABASE_URL="postgresql+psycopg://postgres:postgres@localhost:5432/caihub"
alembic upgrade head
uvicorn app.main:app --reload
```

### 本地快速方式

仅用于测试型环境：

```bash
export CAIHUB_AUTO_CREATE_TABLES=true
uvicorn app.main:app --reload
```

### 规则

- 正式环境不要依赖自动建表
- 表结构变更一律先写 Alembic migration
- 不要在业务代码里偷偷补 `create_all`

## 6. 扩展开发顺序

建议按下面顺序推进：

```text
第一层
  dish
  standard
  production_event

第二层
  store
  operator
  rule_engine

第三层
  event_bus
  analytics_export
  contract_compatibility_test

第四层
  agent runtime
  skill execution runtime
```

## 7. 开发边界约束

### 新增一个业务对象时

至少要补下面几个位置：

```text
model
schema
repository
service
api endpoint
test
如涉及稳定输出，再补 contract
如涉及数据流，再补 mesh metadata
```

### 新增一个稳定数据产品时

至少要补下面几个位置：

```text
app/domains/<domain>/contracts/
app/mesh/contracts.py
app/mesh/flows.py
tests（契约/接口）
docs/数据契约/
```

### 新增一个 agent 或 skill 时

至少要补下面几个位置：

```text
app/agents/registry.py
app/skills/registry.py
docs/智能体/
docs/技能/
必要时补 system/architecture 返回验证
```

## 9. 现阶段判断

当前项目已经具备：

- 可运行
- 可迁移
- 可测试
- 可扩展
- 可治理

但还没有具备：

- 完整关系模型
- 真正事件总线
- 真正规则引擎 DSL
- 真正 agent 编排运行时

所以当前最准确的定义是：

```text
面向平台化演进的第一阶段后端基础仓
```
